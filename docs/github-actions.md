# GitHub Actions

Release Automator includes a composite action plus separate plan, execute, and resume workflows.
Planning prints the complete branch, commit, pull request, checks, merge, cleanup, and release
manifest to the job log and summary. Execution accepts only the exact 64-character frozen plan ID.

## Store credentials as secrets

Open **Settings → Secrets and variables → Actions** and configure these values:

| Name | Location | Purpose |
| --- | --- | --- |
| `OPENAI_API_KEY` | Repository secret | Used only by the planning workflow. |
| `RELEASE_AUTOMATOR_GITHUB_TOKEN` | `release` environment secret | Pushes, opens and merges the PR, reads checks, and creates the release. |

Use a fine-grained personal access token scoped only to the target repository with **Contents:
Read and write**, **Pull requests: Read and write**, and **Checks: Read**. Do not put either value
in workflow inputs, repository variables, configuration files, artifacts, job summaries, or command
arguments. The built-in `${{ github.token }}` is generated by GitHub and does not need to be stored.

Prefer a GitHub App installation token when operating across repositories or when short-lived
credentials are required. Store its private key as an Actions secret, keep the App ID as a variable,
and generate the token in the job. Pass the resulting token to checkout and as `GITHUB_TOKEN`; never
pass the private key to Release Automator.

## Create the approval boundary

Create an environment named `release`, add required reviewers, enable **Prevent self-review** when
available, and add `RELEASE_AUTOMATOR_GITHUB_TOKEN` to that environment. GitHub does not expose the
environment secret until the job is approved. Reviewers should compare the full plan ID and
manifest in the planning run before approving execution.

The bundled workflows are:

- `release-automator-plan.yml`: reconstructs changes from a trusted, full commit SHA, creates the
  portable bundle, prints the manifest, and uploads a 30-day artifact.
- `release-automator-execute.yml`: downloads that artifact by planning run ID, requires the full
  plan ID again, and waits at the `release` environment gate before making writes.
- `release-automator-resume.yml`: restores state from a failed execute/resume run without repeating
  completed remote operations.

Run **Plan a release** from the Actions tab with a repository commit SHA and newline-delimited paths.
After review, run **Execute an approved release plan** with the planning run ID and exact plan ID.
If execution fails after approval, use its run ID with **Resume a release plan**.

## Use the composite action elsewhere

Pin the action to a complete commit SHA, pass ordinary settings through `with`, and pass credentials
only through `env` from named secrets:

```yaml
- uses: your-org/release-automator@0123456789abcdef0123456789abcdef01234567
  with:
    mode: plan
    include-paths: |
      src
      tests
    config-path: release-automator.toml
  env:
    GITHUB_TOKEN: ${{ github.token }}
    OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

Do not use `secrets: inherit`, do not trigger a secret-bearing plan with `pull_request_target`, and
do not plan unreviewed commits. Configured validation commands execute proposed repository code in
the planning job. Fork pull requests do not receive repository secrets and should be validated by
ordinary CI before a maintainer starts the manual planning workflow.

Portable bundles contain the current bytes of included files and may contain private source code.
They intentionally exclude the previous contents of deleted files, but artifact access and
retention should still follow the repository's confidentiality policy.
